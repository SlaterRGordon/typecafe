# Branch Plan: Upgrade Prisma To v7

## Summary

Upgrade TypeCafe from Prisma `4.16.2` to Prisma `7.8.x` on the `64-prisma-upgrade` branch while preserving the current PostgreSQL schema, NextAuth v4 Prisma adapter behavior, tRPC routers, Learn progress, practice stats, score creation, and sitemap generation.

This branch should only change Prisma/server-adjacent code needed for the Prisma 7 upgrade. Do not migrate Auth.js v5, change package managers, switch away from Pages Router, or create destructive database migrations.

Reference:

- [Prisma ORM v7 upgrade guide](https://www.prisma.io/docs/guides/upgrade-prisma-orm/v7)

## Current Baseline

- Branch: `64-prisma-upgrade`
- Starting Prisma CLI: `4.16.2`
- Starting Prisma Client: `4.16.2`
- Target Prisma CLI and Client: `7.8.x`
- Runtime: Node `24.16.0` globally, matching the repo's Node 24 target
- Database: PostgreSQL on Neon
- Schema relation mode: `relationMode = "prisma"`
- NextAuth adapter: `@next-auth/prisma-adapter`

## Goals

- Upgrade `prisma` and `@prisma/client` together to the current stable v7 release.
- Add Prisma 7 config through `prisma.config.ts`.
- Remove `url` from `prisma/schema.prisma` datasource and keep migrations pointed at `DATABASE_URL`.
- Switch to the new `prisma-client` generator with an explicit generated output path.
- Use the PostgreSQL driver adapter through `@prisma/adapter-pg`.
- Update Prisma imports to the generated client path.
- Keep auth, users, tests, practice stats, Learn progress, colors, and sitemap flows working.
- Confirm no accidental migration is generated.

## Non-Goals

- Do not change the database schema unless Prisma 7 requires a mechanical compatibility fix.
- Do not run `prisma migrate dev` against production data.
- Do not apply destructive migration changes.
- Do not migrate NextAuth v4 to Auth.js v5 in this branch.
- Do not combine Tailwind/DaisyUI, Prettier, or unrelated TypeScript major upgrades with this work.

## Required Prisma 7 Changes

- Node must be `20.19+`, `22.12+`, or `24+`; use Node 24 for this repo.
- `prisma.config.ts` is the source for CLI datasource configuration.
- Prisma schema datasource blocks no longer support `url`.
- Prisma Client generation requires an explicit `output`.
- App code imports Prisma Client and model types from the generated output path.
- Runtime database access uses a driver adapter. For PostgreSQL, use `@prisma/adapter-pg`.
- Generated client output is ignored by git and regenerated by `postinstall`.

## Phase 1: Baseline

Record the current Prisma state before package changes.

- Run `npm ls prisma @prisma/client @next-auth/prisma-adapter --depth=0`.
- Run `npx prisma -v`.
- Run `npx prisma validate`.
- Run `npx prisma generate`.
- Run `npm run lint`.
- Run `npx tsc --noEmit --pretty false`.
- Run `npm run build:check`.
- Run `npx prisma migrate status` against the safe development database.

Expected output:

- Baseline commands pass or pre-existing failures are documented before upgrading.
- No schema or migration files are changed during baseline.

## Phase 2: Package Upgrade

Upgrade Prisma packages together.

- Update `prisma` and `@prisma/client` to Prisma 7.
- Add `@prisma/adapter-pg`.
- Add `pg`.
- Add `dotenv` for Prisma CLI env loading through `prisma.config.ts`.
- Keep `@next-auth/prisma-adapter` unchanged unless install or runtime compatibility requires a patch.
- Run `npm install`.

Expected commit:

- `feat(deps): upgrade prisma to v7`

## Phase 3: Config And Generated Client

Apply the Prisma 7 config migration.

- Add `prisma.config.ts` at the repo root.
- Configure:
  - `schema: "prisma/schema.prisma"`
  - `migrations.path: "prisma/migrations"`
  - `datasource.url: env("DATABASE_URL")`
- Update `prisma/schema.prisma`:
  - remove datasource `url`
  - change generator provider to `prisma-client`
  - add `output = "../src/generated/prisma"`
- Add generated Prisma client output to `.gitignore`.
- Add generated Prisma client output to ESLint ignores.

Verification:

- `npx prisma validate`
- `npx prisma generate`

## Phase 4: Runtime And Type Fixes

Fix Prisma 7 type/API differences with small server-scoped edits.

Inspect and test:

- `src/server/db.ts`: import generated `PrismaClient`, create `PrismaPg` adapter with `DATABASE_URL`, and pass it to `new PrismaClient`.
- `src/server/auth.ts`: keep NextAuth v4 adapter behavior. Use a narrow compatibility cast if the adapter package is still typed against the default `@prisma/client` output.
- `src/server/api/routers/test.ts`: `findMany`, `groupBy`, `aggregate`, `distinct`, dynamic `orderBy`.
- `src/server/api/routers/practiceStats.ts`: composite `userId_character` upsert.
- `src/server/api/routers/learnProgress.ts`: transactional create/update import.
- `src/server/api/routers/color.ts`: unique-name creation and delete behavior.
- `src/server/api/routers/user.ts`: user create/update/delete and username/email checks.
- `src/pages/sitemap.xml.ts` and `src/generate-sitemap.ts`: use shared adapter-backed Prisma client.
- Client-side Prisma type imports in profile/edit and scores components.

Verification:

- `npm run lint`
- `npx tsc --noEmit --pretty false`
- `npm run build:check`
- `npx prisma generate`

Expected final commits:

## Rollback Strategy

- If install or generation fails, revert the package upgrade commit.
- If Prisma 7 exposes adapter incompatibility, keep the branch open and do not merge partial Prisma package changes.
- If a migration is accidentally generated, review it before committing; discard it if it is not intentional.
- If auth adapter behavior changes unexpectedly, keep NextAuth v4 pinned and investigate adapter compatibility before considering Auth.js v5.
